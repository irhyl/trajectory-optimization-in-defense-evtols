PLANNING LAYER: Multi-Objective Trajectory Optimization¶

1. Introduction¶

This notebook implements the Planning Layer of the trajectory optimization system for defense eVTOL aircraft. The planning layer is responsible for computing optimal flight paths from the perception data, balancing multiple competing objectives to satisfy mission-specific requirements.

1.1 Conceptual Overview¶

The path planning problem for autonomous aircraft shares fundamental similarities with everyday navigation decisions. Consider how modern navigation systems present route options for a road journey:

  • "Fastest route (25 minutes) — uses toll roads"
  • "Shortest route (15 miles) — through city traffic"
  • "Avoid highways (40 minutes) — scenic route"

The planning layer performs an analogous function for eVTOL operations, but with significantly greater complexity. Rather than optimizing a single objective (time or distance), it must simultaneously balance three competing objectives while respecting the physical constraints of the aircraft and operational environment.

1.2 The Multi-Objective Optimization Challenge¶

Every trajectory involves inherent trade-offs between competing objectives. The three primary objectives for defense eVTOL operations are:

Objective Description Trade-off Implication
Time Minimization Reduce flight duration Direct routes may traverse high-threat areas
Energy Efficiency Minimize battery consumption Optimal paths may require longer flight times
Survivability Minimize detection probability Safe corridors often require significant detours

A fundamental principle of multi-objective optimization is that no single path can simultaneously optimize all objectives. A perfectly safe route avoiding all threats will inevitably be longer and consume more energy. A direct, energy-efficient path will likely traverse areas of elevated threat. The planning layer generates a portfolio of optimal solutions representing different trade-off positions.

1.3 The Three-Stage Planning Process¶

The planning layer operates through three sequential stages:

Stage 1: Path Generation Multiple path-finding algorithms with different search strategies are employed to generate candidate trajectories:

  • A* — Heuristic-guided optimal search (analogous to an experienced navigator taking direct routes)
  • Dijkstra — Exhaustive optimal search (guarantees optimality through comprehensive exploration)
  • Theta* — Any-angle pathfinding (produces smoother, more natural flight paths)
  • RRT* — Sampling-based planning (effective for complex, high-dimensional configuration spaces)

Stage 2: Multi-Objective Evaluation Each candidate path is evaluated against all three objectives. Different weighting schemes are applied to generate solutions optimized for different mission profiles (stealth-priority, speed-priority, efficiency-priority, balanced).

Stage 3: Trajectory Smoothing Raw grid-based paths contain discrete waypoints with sharp angular transitions that are infeasible for actual flight. Cubic spline interpolation transforms these into continuous, flyable trajectories with smooth heading changes.

1.4 Pareto Optimality and Trade-off Analysis¶

When optimizing multiple objectives simultaneously, the concept of Pareto optimality becomes essential. A solution is Pareto-optimal if no other solution can improve one objective without degrading another.

Consider an analogy from consumer decision-making: when purchasing a laptop, one might compare:

  • Laptop A: High performance, high price
  • Laptop B: Low performance, low price
  • Laptop C: Moderate performance, moderate price
  • Laptop D: Low performance, high price

Laptops A, B, and C represent Pareto-optimal solutions—improving one attribute (price or performance) necessarily worsens the other. Laptop D is dominated because Laptop C offers better performance at lower cost.

The planning layer identifies the Pareto frontier of trajectory solutions, providing decision-makers with a clear visualization of available trade-offs. The final trajectory selection is a mission-specific decision based on operational priorities.

5. Path Planning Algorithm Suite¶

5.1 Algorithm Classification and Comparative Analysis¶

This section demonstrates the four fundamental path planning algorithms implemented in our system. Each algorithm possesses distinct computational characteristics and is suited for specific operational scenarios.

5.1.1 Theoretical Framework¶

Algorithm Selection Rationale:

  1. A* (A-Star) Algorithm — The gold standard for heuristic-guided search

    • Guarantees optimal paths when using admissible heuristics
    • Computational complexity: O(b^d) where b is branching factor, d is solution depth
    • Best suited for: Structured grid environments with predictable obstacles

  2. Dijkstra's Algorithm — Exhaustive shortest-path computation

    • Guaranteed optimal paths through systematic exploration
    • Computational complexity: O((V + E) log V) with priority queue implementation
    • Best suited for: Scenarios requiring absolute optimality guarantees

  3. Theta* Algorithm — Any-angle path optimization

    • Produces smoother, more realistic trajectories
    • Allows line-of-sight connections between non-adjacent nodes
    • Best suited for: Applications where path smoothness impacts energy efficiency

  4. RRT* (Rapidly-exploring Random Tree Star) — Probabilistic sampling-based planning

    • Excels in high-dimensional configuration spaces
    • Asymptotically optimal with rewiring capability
    • Best suited for: Complex, cluttered environments with many constraints
5.1.2 Intuitive Comparison: The City Navigation Analogy¶

Consider the challenge of navigating a complex urban environment from your residence to a destination across the city:

Dijkstra Systematic exploration of every possible route Examines all options exhaustively; guaranteed optimal but computationally intensive

class AStarPlanner:
    """A* pathfinding algorithm with multi-objective cost weighting"""
    
    def __init__(self, terrain, threat_map, grid_size=GRID_SIZE, cell_size=CELL_SIZE):
        self.terrain = terrain
        self.threat_map = threat_map
        self.grid_size = grid_size
        self.cell_size = cell_size
    
    def heuristic(self, pos, goal):
        """Euclidean distance heuristic (in meters)"""
        dy = (pos[0] - goal[0]) * self.cell_size
        dx = (pos[1] - goal[1]) * self.cell_size
        return np.sqrt(dy**2 + dx**2)
    
    def get_neighbors(self, pos):
        """Get 8 neighbors of current position"""
        neighbors = []
        for dy in [-1, 0, 1]:
            for dx in [-1, 0, 1]:
                if dy == 0 and dx == 0:
                    continue
                ny, nx = pos[0] + dy, pos[1] + dx
                if 0 <= ny < self.grid_size and 0 <= nx < self.grid_size:
                    neighbors.append((ny, nx))
        return neighbors
    
    def edge_cost(self, pos1, pos2, weights):
        """Compute multi-objective edge cost"""
        w_e, w_r, w_t = weights
        
        # Distance (in meters)
        dy = (pos2[0] - pos1[0]) * self.cell_size
        dx = (pos2[1] - pos1[1]) * self.cell_size
        dist_m = np.sqrt(dy**2 + dx**2)
        
        # Energy: elevation climb + horizontal distance
        elev_gain = max(0, self.terrain[int(pos2[0]), int(pos2[1])] - 
                           self.terrain[int(pos1[0]), int(pos1[1])])
        energy_cost = elev_gain * 2 + (dist_m / 1000) * 5
        
        # Risk: mean threat level
        risk_cost = (self.threat_map[int(pos1[0]), int(pos1[1])] + 
                     self.threat_map[int(pos2[0]), int(pos2[1])]) / 2 * 1000
        
        # Time: distance / cruise speed
        time_cost = dist_m / 15
        
        return w_e * energy_cost + w_r * risk_cost + w_t * time_cost
    
    def plan(self, start, goal, weights):
        """Execute A* search, return path and total cost"""
        open_set = {start}
        came_from = {}
        g_score = {start: 0}
        f_score = {start: self.heuristic(start, goal)}
        closed_set = set()
        
        while open_set:
            # Node with lowest f(n)
            current = min(open_set, key=lambda x: f_score[x])
            
            if current == goal:
                # Reconstruct path
                path = [current]
                while current in came_from:
                    current = came_from[current]
                    path.append(current)
                path.reverse()
                return path, g_score[goal]
            
            open_set.remove(current)
            closed_set.add(current)
            
            for neighbor in self.get_neighbors(current):
                if neighbor in closed_set:
                    continue
                
                tentative_g = g_score[current] + self.edge_cost(current, neighbor, weights)
                
                if neighbor not in g_score or tentative_g < g_score[neighbor]:
                    came_from[neighbor] = current
                    g_score[neighbor] = tentative_g
                    f_score[neighbor] = tentative_g + self.heuristic(neighbor, goal)
                    open_set.add(neighbor)
        
        return None, float('inf')  # No path found

5.3 Comparative Analysis and Algorithm Selection Criteria¶

The preceding visualizations demonstrate the distinctive characteristics of each planning algorithm. Key observations for algorithm selection:

Dijkstra's Algorithm: Exhibits the most extensive exploration pattern (largest colored region), reflecting its exhaustive search nature. While guaranteeing optimality, this computational overhead may be prohibitive for real-time applications.

A* Algorithm: Demonstrates focused, heuristic-guided search with significantly reduced exploration. The Euclidean distance heuristic effectively prunes unpromising search branches while maintaining optimality guarantees.

Theta* Algorithm: Produces visually smoother trajectories by eliminating unnecessary waypoints. The line-of-sight optimization reduces total path length and improves trajectory continuity—critical factors for eVTOL energy efficiency.

RRT* Algorithm: Shows characteristic random sampling behavior. Initial solutions may be suboptimal, but the rewiring mechanism progressively improves path quality toward asymptotic optimality.

        """Compute multi-objective edge cost"""
        w_e, w_r, w_t = weights
        
        dy = (pos2[0] - pos1[0]) * self.cell_size
        dx = (pos2[1] - pos1[1]) * self.cell_size
        dist_m = np.sqrt(dy**2 + dx**2)
        
        elev_gain = max(0, self.terrain[int(pos2[0]), int(pos2[1])] -

        energy_cost = elev_gain * 2 + (dist_m / 1000) * 5
        
        risk_cost = (self.threat_map[int(pos1[0]), int(pos1[1])] + 
                     self.threat_map[int(pos2[0]), int(pos2[1])]) / 2 * 1000
        
        time_cost = dist_m / 15
        
        return w_e * energy_cost + w_r * risk_cost + w_t * time_cost
    
    def plan(self, start, goal, weights):
        """Execute Dijkstra search"""
        # Initialize distances
        dist_dict = {}
        for i in range(self.grid_size):
            for j in range(self.grid_size):
                dist_dict[(i, j)] = float('inf')
        dist_dict[start] = 0
        
        unvisited = set((i, j) for i in range(self.grid_size) for j in range(self.grid_size))
        came_from = {}
        
        while unvisited:
            # Node with minimum distance
            current = min(unvisited, key=lambda x: dist_dict[x])
            
            if dist_dict[current] == float('inf'):
                break  # No more reachable nodes
            
            if current == goal:
                # Reconstruct path
                path = [current]
                while current in came_from:
                    current = came_from[current]
                    path.append(current)
                path.reverse()
                return path, dist_dict[goal]
            
            unvisited.remove(current)
            
            for neighbor in self.get_neighbors(current):
                if neighbor not in unvisited:
                    continue
                
                alt = dist_dict[current] + self.edge_cost(current, neighbor, weights)
                
                if alt < dist_dict[neighbor]:
                    dist_dict[neighbor] = alt
                    came_from[neighbor] = current
        
        return None, float('inf')

class RRTPlanner:
    """RRT* sampling-based motion planning"""
    
    def __init__(self, terrain, threat_map, grid_size=GRID_SIZE, cell_size=CELL_SIZE, samples=300):
        self.terrain = terrain
        self.threat_map = threat_map
        self.grid_size = grid_size
        self.cell_size = cell_size
        self.samples = samples
    
    def sample(self):
        """Random cell sample"""
        return (np.random.randint(0, self.grid_size), np.random.randint(0, self.grid_size))
    
    def nearest(self, tree, sample):
        """Find nearest tree node (Euclidean distance)"""
        dists = [np.sqrt((n[0]-sample[0])**2 + (n[1]-sample[1])**2) for n in tree]

        energy_cost = elev_gain * 2 + (dist_m / 1000) * 5
        
        risk_cost = (self.threat_map[int(pos1[0]), int(pos1[1])] + 
                     self.threat_map[int(pos2[0]), int(pos2[1])]) / 2 * 1000
        
        time_cost = dist_m / 15
        
        return w_e * energy_cost + w_r * risk_cost + w_t * time_cost
    
    def plan(self, start, goal, weights):
        """Execute RRT* search"""
        tree = [start]
        costs = {start: 0}
        parents = {start: None}
        
        for iteration in range(self.samples):
            # Goal biasing: 10% chance to sample goal
            sample = goal if np.random.random() < 0.1 else self.sample()
            nearest = self.nearest(tree, sample)
            cost = costs[nearest] + self.edge_cost(nearest, sample, weights)
            
            tree.append(sample)
            costs[sample] = cost
            parents[sample] = nearest
            
            # Check if goal reached (within 5 cells)
            if np.sqrt((sample[0]-goal[0])**2 + (sample[1]-goal[1])**2) < 5:
                # Reconstruct path
                path = [goal]
                curr = sample
                while curr is not None:
                    path.append(curr)
                    curr = parents[curr]
                path.reverse()
                return path, cost
        
        return None, float('inf')

6. Multi-Objective Optimization: Balancing Competing Mission Requirements¶

6.1 The Fundamental Challenge of Conflicting Objectives¶

In single-objective optimization, success criteria are unambiguous: minimize cost, maximize efficiency, or optimize a single metric. Multi-objective optimization presents a fundamentally different challenge—multiple objectives that cannot be simultaneously minimized.

Illustrative Example: Consider two candidate trajectories with the following performance metrics:

Trajectory Energy (kWh) Duration (min) Detection Probability
Path A 45 30 0.20
Path B 50 25 0.15

Which path is superior? The answer depends entirely on mission priorities:

  • For battery-critical operations → Path A (11% lower energy consumption)
  • For time-sensitive missions → Path B (17% faster completion)
  • For high-threat environments → Path B (25% lower detection probability)

This example illustrates a fundamental principle: no universally optimal solution exists when objectives conflict. The "best" trajectory is context-dependent, determined by mission-specific priorities.

6.2 Detailed Objective Analysis¶

6.2.1 Objective 1: Energy Consumption (Battery Usage)¶

Metric Definition: Total electrical energy expended during flight, measured in kilowatt-hours (kWh).

Computational Model: $$E_{total} = E_{cruise} + E_{climb} + E_{overhead}$$

Where:

  • $E_{cruise}$ = horizontal distance × 5 kWh/km (aerodynamic power for forward flight)
  • $E_{climb}$ = elevation gain × 0.01 kWh/m (work against gravitational potential)
  • $E_{overhead}$ = 2 kWh (fixed consumption for avionics, thermal management, control systems)

Operational Significance:

  • Range Limitation: eVTOL aircraft operate on finite battery capacity, unlike conventionally-fueled aircraft that can carry additional fuel
  • No In-Flight Recharging: Mission must complete within single charge cycle
  • Safety Margins: Prudent operations maintain ≥5% battery reserve for contingencies

Representative Values:

  • Small eVTOL (2-passenger): ~50 kWh battery capacity, ~100 km operational range
  • Mission consumption: 30–60 kWh depending on trajectory and conditions
6.2.2 Objective 2: Flight Duration (Mission Time)¶

Metric Definition: Total elapsed time from departure to destination arrival, measured in minutes.

Computational Model: $$T = \frac{d_{path}}{v_{cruise}} = \frac{d_{path}}{15 \text{ m/s}}$$

This simplified model assumes constant cruise velocity, which is reasonable as cruising dominates flight duration. Actual operations include additional time for takeoff (1–2 min), landing (2–3 min), and turning maneuvers.

Operational Significance:

  • Mission Windows: Intelligence/surveillance operations often have constrained time windows (30–60 minutes of daylight)
  • Urgency Requirements: Emergency response missions prioritize minimum time-to-target
  • Coordination: Ground teams must synchronize with expected arrival times
  • Range Coupling: Longer flight times directly reduce effective range
6.2.3 Objective 3: Detection Risk (Threat Exposure)¶

Metric Definition: Mean probability of detection by adversarial sensors along the trajectory, expressed as a probability in [0, 1].

Computational Model: $$R = \frac{1}{N} \sum_{i=1}^{N} P_{detect}(waypoint_i)$$

Detection probability at each waypoint is computed using a sigmoid model centered on known threat positions: $$P_{detect}(d) = \frac{1}{1 + e^{(d - R_{eff}) / k}}$$

where $d$ is distance from the threat, $R_{eff}$ is effective detection range, and $k$ is a steepness parameter.

Threat Composition:

  • Radar Systems: Long-range detection (30+ km), detects radar cross-section
  • SAM Batteries: Medium-range engagement capability (15–20 km)
  • Visual Observation: Short-range detection (<5 km) by personnel or cameras

Operational Significance:

  • Survivability: High detection probability may result in engagement by air defense systems
  • Mission Success: Detection compromises element of surprise for tactical operations
  • Asset Value: High-value platforms cannot be risked in high-threat corridors

6.3 The Weighting Framework: Encoding Mission Priorities¶

Rather than solving three independent optimization problems, we formulate a weighted aggregation:

$$J_{weighted}(path) = w_E \cdot \hat{E}(path) + w_R \cdot \hat{R}(path) + w_T \cdot \hat{T}(path)$$

Where:

  • $(w_E, w_R, w_T)$ represents the mission priority weight vector
  • $\hat{E}, \hat{R}, \hat{T}$ are normalized objective values (scaled to [0, 1])
  • Weights satisfy the constraint: $w_E + w_R + w_T = 1$

Normalization Requirement: Raw objectives have different units (kWh, minutes, probability). Normalization ensures commensurability: $$\hat{O} = \frac{O - O_{min}}{O_{max} - O_{min}}$$

## Define metrics computation function
def compute_metrics(path, terrain, threat_map):
    """Compute energy, time, and risk metrics for a path"""
    if not path:
        return None
    
    path_arr = np.array(path)
    
    # Distance computation
    distances = np.sqrt(np.sum(np.diff(path_arr, axis=0)**2, axis=1)) * 50  # cell_size=50m
    total_dist = np.sum(distances)
    
    # Elevation profile
    elevations = np.array([terrain[int(p[0]), int(p[1])] for p in path])
    elev_gain = np.sum(np.maximum(0, np.diff(elevations)))
    
    # Energy: distance + climb + fixed overhead
    energy = total_dist / 1000 * 5 + elev_gain * 0.01 + 2
    
    # Time: distance / cruise speed (15 m/s)
    flight_time = total_dist / (15 * 60)
    
    # Risk: mean threat probability along path
    threats = np.array([threat_map[int(p[0]), int(p[1])] for p in path])
    risk = np.mean(threats)
    
    return {
        'distance_km': total_dist/1000,
        'energy_kwh': energy,
        'time_min': flight_time,
        'risk_pd': risk,
        'waypoints': len(path),
        'elevation_gain_m': elev_gain
    }

7. Pareto Frontier Analysis: Identifying Optimal Trade-offs¶

7.1 The Fundamental Challenge: No Single Optimal Solution¶

In single-objective optimization, the concept of optimality is unambiguous—there exists a unique solution that minimizes (or maximizes) the objective function. Multi-objective optimization presents a fundamentally different paradigm: when objectives conflict, no single solution can simultaneously optimize all criteria.

Illustrative Example:

Consider the challenge of selecting a trajectory when you seek to:

  • Maximize fuel efficiency (extend operational range)
  • Minimize flight duration (complete mission rapidly)
  • Minimize detection probability (ensure survivability)

These objectives exhibit inherent conflict:

  • Energy-efficient paths often require longer detours to exploit favorable terrain and wind conditions
  • Time-optimal paths proceed via direct routes that may traverse monitored corridors
  • Low-risk paths circumnavigate threat zones, increasing both distance and duration

The Fundamental Insight: Improving performance on one objective typically degrades performance on another. The notion of a "best" solution becomes context-dependent, determined by mission-specific priorities rather than algorithm output.

7.2 Pareto Optimality: Formal Definition¶

The concept of Pareto optimality provides a rigorous framework for comparing solutions when objectives conflict.

Dominance Relation:

Solution A dominates Solution B if and only if:

  1. A is no worse than B in all objectives, AND
  2. A is strictly better than B in at least one objective

Non-Dominated Solutions:

A solution is non-dominated (or Pareto-optimal) if no other feasible solution dominates it. The set of all non-dominated solutions constitutes the Pareto frontier.

Practical Interpretation:

Solutions on the Pareto frontier represent the "best possible" trade-offs—moving from one frontier solution to another necessarily involves sacrificing performance on at least one objective to gain on another. Solutions not on the frontier are suboptimal and should be discarded.

7.3 Conceptual Analogy: Consumer Decision-Making¶

The Pareto frontier concept is readily understood through a consumer decision-making analogy. Consider evaluating four vehicles on three criteria:

Vehicle Price ($k) Top Speed (mph) Comfort Rating
Sports Car 50 200 Low
Sedan 35 120 High
SUV 45 100 High
Luxury 80 130 Exceptional

Dominance Analysis:

  • Sports Car vs. Sedan: Sports Car is faster but more expensive with lower comfort—neither dominates
  • SUV vs. Sedan: Trade-off between cargo capacity and speed—neither dominates
  • Luxury vs. Sedan: Luxury excels in comfort but at substantially higher price—trade-off exists

Conclusion: All four vehicles represent Pareto-optimal choices for their respective priority configurations. The "best" vehicle depends entirely on the purchaser's priorities (performance, economy, luxury).

7.4 Application to eVTOL Trajectory Selection¶

When applied to our 16-solution portfolio, Pareto analysis yields:

Input: 16 solutions with three objective values each (Energy, Time, Risk)

Process: Identify all non-dominated solutions

Output: Pareto frontier containing 3–6 solutions (typically)

Decision Framework:

  1. Discard all dominated solutions (strictly inferior alternatives)
  2. Examine only frontier solutions (optimal trade-offs)
  3. Select based on mission-specific priorities

7.5 The Three Fundamental Trade-offs¶

Energy-Optimized Path Time-Optimized Path

7.6 Mathematical Formulation¶

Problem Formulation:

Given solution set $S = \{s_1, s_2, \ldots, s_{16}\}$ where each solution has objective vector: $$s_i = (E_i, T_i, R_i) = (\text{Energy}_i, \text{Time}_i, \text{Risk}_i)$$

Dominance Condition:

Solution $s_i$ dominates $s_j$ (denoted $s_i \prec s_j$) if and only if: $$\forall k \in \{E, T, R\}: s_i^k \leq s_j^k \quad \text{AND} \quad \exists k: s_i^k < s_j^k$$

Pareto Frontier: $$\mathcal{F} = \{s \in S : \nexists s' \in S \text{ such that } s' \prec s\}$$

7.7 Decision Framework for Mission Planning¶

Once the Pareto frontier is identified, trajectory selection proceeds according to mission requirements:

Mission Category Primary Objective Recommended Selection
Strategic Long-Range Energy Minimization Frontier solution with lowest $E$
Tactical Insertion Risk Minimization Frontier solution with lowest $R$
Emergency Response Time Minimization Frontier solution with lowest $T$
General Operations Balanced Frontier solution nearest to normalized origin

Trade-off Quantification:

When comparing two frontier solutions, the explicit cost of transitioning can be computed:

  • $\Delta E$: Additional energy required
  • $\Delta T$: Additional time required
  • $\Delta R$: Additional risk incurred

This quantification enables informed decision-making: "Is the 15% reduction in risk worth the 20% increase in energy consumption?"

def is_dominated(sol, all_sols, objectives=['Energy_kWh', 'Time_min', 'Risk_Pd']):
    """Check if a solution is dominated by any other solution (minimize all)"""
    for other in all_sols:
        if other is sol:
            continue
        # Check if other dominates sol: better in all objectives
        dominates = all(other[obj] <= sol[obj] for obj in objectives)
        strictly_better = any(other[obj] < sol[obj] for obj in objectives)
        if dominates and strictly_better:
            return True
    return False

8. Trajectory Smoothing: From Discrete Waypoints to Flight-Ready Paths¶

8.1 The Grid-to-Flight Transformation Challenge¶

Path planning algorithms operate on discrete grid representations, producing waypoint sequences that, while mathematically optimal, are not directly executable by aircraft flight control systems. This section addresses the critical transformation from discrete computational outputs to continuous, flyable trajectories.

The Fundamental Problem:

Grid-based algorithms produce paths constrained to cell centers and 8-directional movements:

Grid Path (Discrete):              Required Flight Path (Continuous):

    *─*─*                              ╭──────────╮
     ╲ ╲ ╲                            ╱            ╲
      ╲ ╲ ╲         →                ╱              ╲
       ╲ ╲ ╲                        ╱                ╲
        *─*─*                      ╰──────────────────╯

Characteristics of Grid Paths:

  • Angular discontinuities: 45° or 90° heading changes at each waypoint
  • Discrete spacing: Waypoints separated by fixed cell dimensions (50m)
  • Control incompatibility: Autopilot systems cannot execute instantaneous heading changes
  • Distance inflation: Zigzag patterns increase actual flight distance

8.2 Cubic Spline Interpolation: The Smoothing Solution¶

Cubic spline interpolation transforms discrete waypoint sequences into smooth, continuous curves that preserve the essential path geometry while eliminating discontinuities.

Properties of Cubic Splines:

  • Exact waypoint passage: The interpolated curve passes through all original waypoints
  • Continuity guarantees: Position, velocity, and acceleration are continuous (C² continuity)
  • Curvature control: Smooth transitions without abrupt directional changes
  • Minimal deviation: The smoothed path closely approximates the original intent

Mathematical Formulation:

For a trajectory segment between consecutive waypoints $i$ and $i+1$, the position is given by a cubic polynomial:

$$\mathbf{r}(t) = \mathbf{a}_i t^3 + \mathbf{b}_i t^2 + \mathbf{c}_i t + \mathbf{d}_i, \quad t \in [0, 1]$$

The coefficients $(\mathbf{a}_i, \mathbf{b}_i, \mathbf{c}_i, \mathbf{d}_i)$ are determined by enforcing:

  • Position continuity at waypoints
  • First derivative (velocity) continuity
  • Second derivative (acceleration) continuity
  • Boundary conditions at path endpoints

8.3 The Three-Stage Smoothing Process¶

Stage 1: Coordinate Transformation

Convert grid cell indices to metric coordinates: $$\mathbf{p}_{metric} = \mathbf{p}_{grid} \times \Delta_{cell}$$

where $\Delta_{cell}$ is the grid resolution (50 meters in our implementation).

Stage 2: Spline Construction

Construct independent cubic splines for each coordinate dimension:

  • $\mathcal{S}_x(s)$: Easting coordinate as function of path parameter
  • $\mathcal{S}_y(s)$: Northing coordinate as function of path parameter
  • $\mathcal{S}_z(s)$: Altitude coordinate as function of path parameter

Stage 3: Waypoint Densification

Evaluate the splines at fine intervals to generate a dense waypoint sequence:

  • Original path: ~100 waypoints at 50m spacing
  • Densified path: ~2,500 waypoints at 2m spacing
  • Result: 25× increase in guidance resolution

8.4 Elevation and Threat Profile Integration¶

The smoothed trajectory must be augmented with elevation and threat information sampled from the environmental models.

Elevation Embedding:

For each interpolated waypoint at position $(x, y)$:

  1. Locate the surrounding grid cells
  2. Apply bilinear interpolation to terrain elevation values
  3. Assign continuous altitude profile respecting terrain clearance

Threat Annotation:

For each interpolated waypoint:

  1. Evaluate threat map at the interpolated position
  2. Record instantaneous detection probability
  3. Enable continuous risk profile visualization

8.5 Flight Dynamics Validation¶

Smoothed trajectories must satisfy aircraft kinematic constraints to ensure executability.

Curvature Constraints:

The minimum turn radius for an eVTOL at cruise velocity $v$ with maximum bank angle $\phi$ is: $$R_{min} = \frac{v^2}{g \tan(\phi)}$$

For typical parameters ($v = 30$ m/s, $\phi = 30°$): $$R_{min} \approx 150 \text{ meters}$$

Spline curvature must not exceed $\kappa_{max} = 1/R_{min}$ at any point.

Rate Constraints:

  • Maximum heading rate: ~30°/second (typical eVTOL capability)
  • Maximum climb/descent rate: 5–10 m/s (energy-limited)
  • Maximum acceleration: 0.3g (passenger comfort)

8.6 Comparative Analysis: Grid Path vs. Smoothed Trajectory¶

Characteristic Grid Path Smoothed Trajectory
Waypoint Count 100–200 500–2,500
Waypoint Spacing 50m (fixed) 2–5m (continuous)
Heading Changes 45° discrete jumps <1° per waypoint

8.8 Quality Assurance Verification¶

Post-smoothing validation confirms trajectory integrity:

  1. Collision Clearance: All waypoints maintain minimum separation from terrain and obstacles
  2. Threat Exposure: Smoothed path does not drift into high-threat regions
  3. Kinematic Feasibility: Curvature and rate constraints satisfied throughout
  4. Metric Preservation: Energy, time, and risk metrics remain within acceptable tolerance of original path values

Conclusion: Trajectory smoothing is an essential processing stage that transforms computationally-optimal but physically-infeasible grid paths into flight-ready trajectories suitable for autonomous eVTOL operations.
